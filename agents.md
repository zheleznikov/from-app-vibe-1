Ты — senior full-stack разработчик. Создай полностью работоспособное учебное full-stack приложение «Мини-анкета».

Приложение должно состоять из:

* frontend на React + TypeScript;
* backend на Node.js + TypeScript;
* Docker Compose для запуска всего приложения одной командой.

## Цель

Нужно создать минимальное, но аккуратно структурированное full-stack приложение, в котором пользователь:

1. открывает страницу с анкетой;
2. получает список вопросов с backend;
3. заполняет ответы;
4. отправляет их на backend;
5. видит сообщение «Спасибо!» после успешной отправки.

Данные должны храниться только в оперативной памяти backend-приложения. Базу данных подключать не нужно.

## Технологический стек

### Frontend

Используй:

* React;
* TypeScript;
* Vite;
* Fetch API для HTTP-запросов;
* обычный CSS или CSS Modules.

Не используй тяжёлые UI-библиотеки и глобальные state-management решения. Для этого приложения достаточно локального состояния React.

### Backend

Используй:

* Node.js;
* TypeScript;
* Express;
* CORS;
* встроенное хранение данных в массиве или Map.

Добавь базовую валидацию входящих данных.

### Инфраструктура

Используй:

* Docker;
* Docker Compose;
* отдельный Dockerfile для frontend;
* отдельный Dockerfile для backend.

Приложение должно запускаться командой:

```bash
docker compose up --build
```

После запуска frontend должен быть доступен в браузере, например:

```text
http://localhost:3000
```

Backend API должен быть доступен, например:

```text
http://localhost:4000
```

## Функциональные требования

### Backend API

Реализуй два обязательных endpoint.

### GET /questions

Возвращает список из 3–5 заранее заданных вопросов анкеты.

Пример ответа:

```json
[
  {
    "id": "name",
    "text": "Как вас зовут?",
    "type": "text",
    "required": true
  },
  {
    "id": "experience",
    "text": "Сколько лет вы занимаетесь разработкой?",
    "type": "number",
    "required": true
  },
  {
    "id": "feedback",
    "text": "Что вы ожидаете от курса?",
    "type": "textarea",
    "required": false
  }
]
```

Допустимые типы вопросов:

* `text`;
* `number`;
* `textarea`.

Можно добавить другие простые типы, но это необязательно.

### POST /answers

Принимает заполненные пользователем ответы.

Пример запроса:

```json
{
  "answers": {
    "name": "Иван",
    "experience": 3,
    "feedback": "Хочу научиться создавать full-stack приложения"
  }
}
```

Backend должен:

1. проверить наличие поля `answers`;
2. проверить, что обязательные вопросы заполнены;
3. сохранить отправленные данные в памяти;
4. добавить уникальный идентификатор и дату отправки;
5. вернуть успешный ответ.

Пример успешного ответа:

```json
{
  "success": true,
  "message": "Ответы сохранены",
  "submissionId": "generated-id"
}
```

При ошибке валидации возвращай HTTP 400 и понятное описание ошибки.

Пример:

```json
{
  "success": false,
  "message": "Не заполнен обязательный вопрос: Как вас зовут?"
}
```

Для необработанных ошибок возвращай HTTP 500.

### Дополнительный endpoint

Для проверки работы приложения можешь добавить:

```text
GET /health
```

Ответ:

```json
{
  "status": "ok"
}
```

Дополнительно можно добавить:

```text
GET /answers
```

Этот endpoint нужен только для локальной проверки сохранённых ответов. В интерфейсе frontend его использовать не обязательно.

## Хранение данных

Сохраняй ответы в памяти backend-приложения, например:

```ts
const submissions: Submission[] = [];
```

После перезапуска backend или Docker-контейнера данные могут удаляться. Это ожидаемое поведение.

Не подключай PostgreSQL, MongoDB, Redis и другие внешние хранилища.

## Требования к frontend

Frontend должен:

1. при загрузке страницы отправлять `GET /questions`;
2. показывать состояние загрузки;
3. показывать ошибку, если вопросы получить не удалось;
4. динамически сформировать форму на основании ответа backend;
5. отображать разные HTML-поля в зависимости от типа вопроса;
6. хранить введённые ответы в состоянии React;
7. проверять обязательные поля перед отправкой;
8. отправлять данные через `POST /answers`;
9. блокировать кнопку отправки во время запроса;
10. показывать сообщение об ошибке при неуспешной отправке;
11. после успешной отправки скрывать форму или очищать её;
12. показывать пользователю сообщение:

```text
Спасибо!
```

Интерфейс должен быть простым, но аккуратным:

* форма расположена по центру страницы;
* у полей есть подписи;
* обязательные поля отмечены;
* кнопка имеет состояния normal и disabled;
* сообщения об ошибках хорошо заметны;
* приложение корректно выглядит на мобильном экране.

## Типизация

Создай явные TypeScript-типы для:

* вопроса анкеты;
* типа вопроса;
* пользовательских ответов;
* отправленной анкеты;
* ответа API;
* ошибки API.

Не используй `any`, кроме ситуаций, где это действительно невозможно избежать.

Frontend и backend могут иметь отдельные файлы типов. Настраивать общий workspace-пакет ради такого небольшого приложения необязательно.

## Работа frontend с backend

Не прописывай адрес backend хаотично в компонентах.

На frontend используй переменную окружения:

```text
VITE_API_URL
```

Например:

```env
VITE_API_URL=http://localhost:4000
```

В Docker Compose настрой эту переменную так, чтобы frontend корректно обращался к backend из браузера.

Учитывай разницу между:

* адресом контейнера внутри Docker-сети;
* адресом API, который использует браузер пользователя.

Поскольку запрос выполняется браузером, значение API URL должно быть доступно с хост-машины, например:

```text
http://localhost:4000
```

На backend настрой CORS для frontend.

## Docker

Создай production-oriented Dockerfile для каждого приложения.

### Backend Dockerfile

Он должен:

1. установить зависимости;
2. скомпилировать TypeScript;
3. запускать скомпилированный JavaScript;
4. слушать `0.0.0.0`, а не только `localhost`.

### Frontend Dockerfile

Предпочтительный вариант:

1. собрать React-приложение через Vite;
2. раздавать статические файлы через Nginx.

Настрой Nginx так, чтобы приложение корректно открывалось при прямом переходе на frontend URL.

Если маршрутизация React Router не используется, сложная SPA-конфигурация не требуется, но стандартный fallback на `index.html` допустим.

### Docker Compose

В `docker-compose.yml` должны быть два сервиса:

* `frontend`;
* `backend`.

Требования:

* backend публикует порт `4000`;
* frontend публикует порт `3000`;
* frontend зависит от backend;
* контейнеры находятся в общей Docker-сети;
* добавь healthcheck для backend, если это не усложняет проект;
* не используй устаревшее поле `version`.

## Обработка ошибок

Предусмотри следующие ситуации:

* backend недоступен;
* `GET /questions` вернул ошибку;
* сервер вернул некорректные данные;
* пользователь не заполнил обязательное поле;
* `POST /answers` вернул HTTP 400;
* произошла внутренняя ошибка backend;
* пользователь несколько раз нажал кнопку отправки.

Все ошибки должны приводить к понятному сообщению в интерфейсе, а не к падению приложения.
